# Pop WiFi MV Studio — 인터랙티브 최종 개발 가이드 (통합본)

## 0. 문서 목적
이 문서는 Pop WiFi MV Studio를 단순한 화면 묶음이 아니라, **PC 전용 인터랙티브 MV 스튜디오**로 완성하기 위한 최종 통합 가이드다.

이 문서는 다음을 동시에 묶는다.
- 기존 MV Studio 개발 지침
- 인트로 프리셋 번들 원칙
- 디자인 보호 원칙
- 모듈화 및 내부 handoff 원칙
- Haven 계열의 프리미엄 인터랙티브 감성 지침
- 가사 마킹 / 노래방 진행 / 랜딩 검수 / 최종 렌더 전 구조 지침

이 문서는 **보이는 화면용 문서가 아니라 내부 개발 기준 문서**다.

---

## 1. 절대 고정 원칙

### 1-1. 지금의 목표
지금의 목표는 프리셋을 한 번에 통합하는 것이 아니라, **MV Studio 플랫폼 구조를 먼저 완성**하는 것이다.

### 1-2. 디자인 보호 원칙
- 기존 디자인의 핵심 톤을 함부로 바꾸지 않는다.
- 기존 레이아웃을 깨는 대수술을 한 번에 하지 않는다.
- 디자인 수정은 단계적으로 한다.
- 보이는 화면에는 개발 문구를 넣지 않는다.
- 내부 지침과 handoff는 반드시 `docs/` 폴더에서 따로 관리한다.

### 1-3. 구조 원칙
- 하나의 거대 파일 금지
- 페이지, 레이어, 상태, 폰트, 프리셋, 랜딩 관련 로직은 모듈로 분리
- Preview / Landing / Final Render는 같은 React 구조를 공유
- 인트로 프리셋은 사용자 기준으로 단일 번들 유지

### 1-4. 레이아웃 원칙
- 2열로 갈 수 있는 것은 2열 유지
- 1열은 좁은 폭에서만 허용
- 세로 누적로 스크롤을 불필요하게 늘리지 않는다
- 핵심 메뉴명은 `...` ellipsis 처리 금지
- 사이드바는 정보 축약용이 아니라 스튜디오 네비게이션이다

## 2. 기술 스택 기준

### 2-1. 프런트엔드
- React 기반
- PC 전용 스튜디오형 UX 우선
- 상태 기반 화면 전환 구조 유지
- 정적 HTML 나열 금지

### 2-2. 렌더 / 후처리
- Remotion: 메인 렌더 구조
- FFmpeg: 후처리
- Whisper Local: 1차 가사 타임 마킹

### 2-3. 폰트 정책
- 시스템 폰트 사용 금지
- 반드시 Google Fonts 기반으로 운영
- 제목 폰트 / 가사 폰트 분리
- 폰트 선택 UI는 Photoshop식 브라우저/모달 패턴 사용
- 메인 화면에 전체 폰트 리스트를 미리 뿌리지 않는다
- 선택 시 로드, 이후 캐시 재사용 구조로 간다
- 가능하면 추후 실제 폰트 파일 다운로드 + 로컬 캐시 폴더 구조까지 확장 가능하게 설계

## 3. 프로젝트 구조 및 데이터 설계 지침

### 3-1. Studio를 툴이 아니라 플랫폼으로 설계
- 프로젝트 설정, 프리셋 데이터, 폰트 리스트, 업로드 정보, 가사 데이터는 각각 독립 모듈로 분리
- 메인 파일 비대화 금지
- 기능이 늘어날수록 모듈 단위로만 확장

### 3-2. 상태 관리 기준
프로젝트 전체를 하나의 상태 모델로 다룬다.
예:
- projectMeta
- projectType (single / playlist)
- projectLanguage
- trackCount
- folderPath
- uploadedAssets
- introPresetState
- lyricsMarkingState
- landingConfig
- renderConfig

### 3-3. 자동 저장 기준
- localStorage 혹은 IndexedDB 기반 자동 저장 포함
- 새로고침 후에도 최근 상태 복원
- 저장 시점은 입력 중 debounce + 명시 저장 둘 다 지원

### 3-4. 프로젝트 생성 UX 기준
프로젝트 페이지에서 반드시 아래를 명확히 처리한다.
- 프로젝트명
- 프로젝트 타입: 싱글 / 플레이리스트
- 언어 규칙
  - 싱글 = 한국어
  - 플레이리스트 = 영어
- 플레이리스트 곡 수 선택
  - 6 / 8 / 10 / 12
- 업로드 위치 안내
  - 단순 설명 문구가 아니라 실제 폴더 트리 시각화 제공
  - 예: `ProjectName/01/audio.mp3`
- 싱글: 단일 파일 업로드 카드
- 플레이리스트: 프로젝트 폴더 업로드 카드

## 4. 인트로 프리셋 구조 원칙

### 4-1. 단일 번들 원칙
인트로 프리셋은 아래가 한 몸이다.
- 사각 프레임
- 제목 디자인
- 좋아요 / 구독 / 알림
- 하단바
- 끝 위젯 애니메이션
- 제목 폰트
- 가사 폰트
- 컬러 variant
- 질감 / 노이즈
- 타이밍 규칙

### 4-2. 사용자 선택 단위
사용자 기준으로는 항상 1개 프리셋을 선택한다.
내부적으로만 다음처럼 분리 가능:
- FrameLayer
- TitleLayer
- ActionLayer
- BottomBarLayer
- KaraokeLyricLayer
- WidgetLayer

### 4-3. 폰트 UX
- 제목 폰트 선택 카드
- 가사 폰트 선택 카드
- 각 카드 클릭 시 모달/드로어 오픈
- 검색 가능
- 실제 폰트 미리보기로 표시
- 선택 즉시 프리뷰 반영

---

## 5. 가사 마킹 및 데이터 무결성 지침

### 5-1. 가사 마킹의 역할
가사 기능은 작사가 아니라 **타이밍 교정 도구**다.
흐름:
1. Whisper 1차 자동 생성
2. 사용자 2차 보정
3. 교정 결과를 렌더 truth로 사용

### 5-2. 정밀 교정 인터페이스
- 오디오 파형(Waveform) 시각화 필수
- 재생, 일시정지, 짧게 뒤로, 짧게 앞으로, 반복 재생 제공
- 현재 줄 활성 강조
- 시작/끝 미세 조정
- 줄 분할 / 줄 합치기 지원

### 5-3. 실시간 검증
- 가사 오버랩 감지
- 한 줄 글자수 초과 감지
- 하단바 가독성 위반 감지
- 충돌 발생 시 즉시 시각적 경고

### 5-4. 진행률 배지
사이드바에 아래 상태를 배지로 표시
- 프로젝트 준비
- 프리셋 준비
- 가사 마킹 (예: 12곡 중 3곡 완료)
- 랜딩 검수
- 최종 렌더 준비

---

## 6. 노래방 진행 구조 원칙

### 6-1. 설정 위치
노래방 진행 스타일 선택 UI는 **인트로 프리셋 화면 안에 둔다.**

### 6-2. 구현 위치
실제 렌더 로직은 별도 레이어로 분리한다.
- `KaraokeLyricLayer`
- `BottomBarLayer`

### 6-3. 추천 구조
- 일반 하단바
- 현재 줄 강조형 (Line Active)
- 단어 채움형 (Word Fill)
- 전체 노래방형 (Full Karaoke)

### 6-4. 초기 구현 추천
초기 1차는 **Line Active + BottomBar 유지형**으로 간다.
복잡도 대비 안정성이 가장 높다.

## 7. Haven 계열 인터랙티브 & 모션 지침

### 7-1. 목표 감성
사용자가 “웹페이지”가 아니라 **고급 설치형 스튜디오**를 쓰는 느낌을 받아야 한다.

### 7-2. 커서 기반 인터랙션
- 버튼 / 카드 마그네틱 hover
- 현재 기능 상태를 커서 보조 텍스트로 표시 가능
  - SELECT
  - OPEN
  - DRAG
  - APPLY

### 7-3. 타이포그래피 리듬감
- 제목 → 패널 → 세부 설정이 순차적으로 나타나는 stagger 모션
- 폰트 변경 시 거칠게 튀지 말고 부드럽게 전환
- 과한 효과보다 정교한 속도감이 중요

### 7-4. 살아있는 질감
- 노이즈는 정지 오버레이가 아니라 미세한 루프 모션 허용
- 포인트 컬러 glow가 은은하게 공간감을 만듦
- 배경은 깊이감이 있어야 하고, 번쩍이는 UI가 되어선 안 됨

### 7-5. 색상 전이
- 프리셋 변경, 컬러 변경, 폰트 변경 시 즉시 튀지 않고 0.35~0.5초 안쪽에서 부드럽게 전이

### 7-6. 페이지 전환
- 메인 프리뷰는 자리를 지키고
- 주변 패널들만 교체되는 방식 유지
- 화면 전체를 갈아엎는 페이지 점프 느낌 금지

## 8. 레이아웃 및 정보 밀도 지침

### 8-1. 사이드바
사이드바는 매우 중요하다.
- 메뉴명 ellipsis 금지
- 최대 2줄 자연 줄바꿈 허용
- 아이콘 + 제목 + 보조 설명 + 상태 배지 구조
- 프로젝트 요약 카드 포함
- 단순한 내비게이션이 아니라 스튜디오의 상태판 역할

### 8-2. 중앙 작업 영역
- 프리뷰는 항상 중심
- 폼이 아니라 “작업 패널”처럼 보여야 함
- 너무 긴 세로 폼 금지
- 여백과 계층감으로 고급스러움 확보

### 8-3. 카드 배치 원칙
- 2열 가능하면 2열 유지
- 3열 가능한 작은 옵션은 3열 허용
- 1열 남발 금지
- 좁은 폭에서만 1열로 무너짐 허용

### 8-4. 랜딩 검수 화면
- 좌측 옵션 영역도 세로로만 늘리지 말고 2열 우선
- 검수 항목, 렌더 옵션, 적용 속성 요약을 효율적으로 배치

## 9. 리액트의 장점을 반드시 살리는 기준

### 9-1. 금지
- HTML을 JSX로 옮긴 듯한 정적 구조
- 모든 UI를 한 번에 뿌리는 방식
- 리스트 미리 전부 렌더
- 스크롤만 길어지는 세로 폼

### 9-2. 필수
- 상태 기반 패널 교체
- 모달/드로어 기반 선택 인터랙션
- 카드 hover 반응
- 선택 즉시 프리뷰 반영
- 페이지 전환 / 패널 전환 / 컬러 전환 / 폰트 전환에 모션 부여
- “쫀득한” 반응과 여유로운 감성 유지

## 13. 최종 선언
이 프로젝트는 단순한 웹 UI가 아니라,
**프리셋 선택, 폰트 선택, 가사 교정, 랜딩 검수, 최종 렌더 준비**를 하나의 흐름으로 묶는 **프리미엄 PC 전용 MV 스튜디오**다.

기술적으로는 데이터 안정성과 모듈화를 확보하고,
감성적으로는 커서와 선택, 전환, 질감, 속도감에 반응하는 부드러운 인터랙션을 심어야 한다.

핵심 문장:

> 기술적으로는 데이터의 안정성을 확보하고,
> 감성적으로는 사용자의 시선과 움직임에 반응하는 부드러운 가속도를 심는다.
> PC 전용 스튜디오인 만큼, 빽빽한 정보들 사이에서도 여유롭고 세련된 인터랙션이 느껴져야 한다.
